# 논리아키텍처설계 가이드

## 목적

유저스토리와 아키텍처 패턴을 기반으로 서비스 간 관계 중심의 논리 아키텍처를 설계함. Context Map 스타일의 컴포넌트 다이어그램으로 시각화하여 이후 HighLevel 아키텍처 정의의 기반을 제공함.

## 방법론

### 1. 준비

- 유저스토리, 아키텍처패턴을 분석 및 이해합니다.
- 핵심솔루션에서 AI 활용이 필요한 영역을 식별합니다.
- 논리아키텍처 다이어그램 예제 참조

### 2. 실행

#### 2.1 논리 아키텍처 설계서 작성

`logical-architecture.md` 파일에 아래 항목을 필수 포함하고 필요 시 항목을 추가합니다.

- **개요**: 설계 원칙, 핵심 컴포넌트 정의
- **서비스 아키텍처**
  - 서비스별 책임
  - 서비스 간 통신 전략
- **AI 서비스 아키텍처** (AI 활용 기능이 있는 경우 필수)
- **주요 사용자 플로우**
- **데이터 흐름 및 캐싱 전략**
- **확장성 및 성능 고려사항**

- **논리아키텍처 다이어그램** (별도 `.mmd` 파일 경로 참조)

#### 2.2 AI 서비스 아키텍처 설계

핵심솔루션 및 유저스토리에서 AI가 핵심 가치를 제공하는 기능이 있으면, AI 서비스를 독립 바운디드 컨텍스트(서비스)로 식별하고 논리 아키텍처에 포함합니다.

> **원칙**: AI가 제품의 핵심 가치 제안(Core Value Proposition)에 해당하면, 논리 아키텍처 설계 시점에 AI 서비스의 경계와 통신을 함께 정의해야 합니다. AI 상세 설계(모델 선정, 프롬프트, 비용 최적화 등)는 별도 Step에서 수행하되, 서비스로서의 위치는 이 단계에서 확정합니다.

**AI 서비스 식별 기준**:

1. **핵심 가치 기여도**: 유저스토리에서 AI가 핵심 기능(자연어 생성, 추천, 분석 등)을 담당하는가?
2. **기술 스택 이질성**: 메인 서비스(Java/Spring 등)와 다른 기술 스택(Python/FastAPI 등)이 필요한가?
3. **독립 스케일링**: AI 워크로드의 리소스 패턴(GPU, 메모리)이 다른 서비스와 다른가?
4. **외부 AI API 의존성**: LLM API(OpenAI, Claude, Gemini 등) 호출이 필요한가?

**AI 서비스 설계 시 필수 포함 항목**:

```
#### AI 서비스 (AI Pipeline / AI Gateway 등)
**책임**: {AI 서비스의 핵심 책임 정의}

**핵심 기능**:
- {AI 기능 1} - {관련 유저스토리 ID}
- {AI 기능 2} - {관련 유저스토리 ID}

**AI 기술 요소**:
- LLM API: {사용할 LLM 제공자 — OpenAI / Claude / Gemini / Azure OpenAI 등}
- AI 프레임워크: {LangChain / LlamaIndex / 직접 구현 등}
- RAG: {사용 여부 및 벡터 DB — Chroma / Pinecone / pgvector 등}
- Function Calling: {도구 호출 필요 여부 및 대상}
- MCP: {Model Context Protocol 활용 여부}

**통신 방식**: {동기/비동기, 호출 서비스 → AI 서비스 방향}
**적용 패턴**: {Circuit Breaker, Cache-Aside, Rate Limiting 등}

**API 경계**:
- {AI API 엔드포인트 목록}

**폴백 전략**:
- {AI 장애 시 대체 동작 — 템플릿 폴백, 캐시 응답, Graceful Degradation 등}
```

**AI 서비스 통신 설계**:

- AI 서비스를 호출하는 서비스(소비자)와의 통신 방식을 명시
- 동기 통신: 사용자 응답에 AI 결과가 즉시 필요한 경우
- 비동기 통신: 배치 처리, 사전 생성 등 시간 여유가 있는 경우
- Circuit Breaker + 폴백: AI API 장애 시 서비스 연속성 보장
- Rate Limiting: LLM API 비용 제어 및 할당량 관리

**다이어그램 반영**:

- AI 서비스를 논리아키텍처 다이어그램에 독립 서비스로 표현
- AI 서비스 → 외부 LLM API 연결선 표현
- AI 서비스를 호출하는 서비스들의 의존 관계 표현
- 캐시 계층(AI 응답 캐시)이 있으면 별도 표현

#### 2.3 논리 아키텍처 다이어그램 작성

`logical-architecture.mmd` 파일에 Mermaid 형식으로 작성합니다.

**통신 전략**:
- **동기 통신**: 즉시 응답이 필요한 단순 조회
- **캐시 우선**: 자주 사용되는 데이터는 캐시에서 직접 읽기
- **비동기 처리**: 외부 API 다중 호출 등 장시간 작업

**다이어그램 작성 원칙**:
- Context Map 스타일로 서비스 내부 구조는 생략하고 서비스 간 관계에 집중
- 클라이언트에서 API Gateway로는 단일 연결선으로 표현
- 사용자 관점의 컴포넌트 다이어그램 작성
- **AI 서비스가 있으면 독립 서비스로 표현하고, 외부 LLM API와의 연결을 명시**

#### 2.4 Mermaid 스크립트 검사 실행

작성 완료 후 'Mermaid문법검사가이드(`{PLUGIN_DIR}/resources/guides/tools/mermaid-guide.md`)'에 준용하여 스크립트를 검증합니다.

### 3. 검토

- 작성원칙 준수 검토
- AI 서비스가 핵심 컴포넌트 테이블에 포함되었는지 확인
- AI 서비스 통신 흐름이 서비스 간 통신 전략에 반영되었는지 확인

## 주의사항

- **유저스토리와 매칭**: 불필요한 추가 설계 금지
- '아키텍처패턴'에 선정된 클라우드 디자인 패턴을 적용하여 설계
- 물리적 배포 구조(인프라, 컨테이너 배치 등)는 이 단계에서 결정하지 않음
- **AI 서비스의 상세 설계(모델 선정, 프롬프트 설계, 비용 최적화)는 이 단계에서 수행하지 않음** — 서비스 경계, 통신 방식, 폴백 전략만 정의
- 설계 공통 원칙: `{PLUGIN_DIR}/resources/guides/design/common-principles.md` 준용
